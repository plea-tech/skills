# 报告的输出格式或者大纲如下:
1. 公司概况
1.1 行业位置&愿景
1.2 规模&营收
1.3 市场情况
1.4产品组合
1.5组织架构
2. 产品分析
2.1 架构&技术分析
2.2 功能分析, 分开通，保障，资源三大块进行分析.
2.3 业务支撑能力, 对电信行业业务的支撑程度
2.4 网络支撑能力, 按网络专业进行分析.
2.5 系统集成能力, 集成第三方系统的能力，包括OOTB适配器, 协议等; 通过标准API被第三方系统集成的能力等.
2.6 可配置性, 系统的配置能力, 尤其是服务目录配置，流程模板配置，资源分配策略配置，激活&指令配置，施工项配置等.
2.7 开放性, 是否有二开工具，是否支持低码/零码开发等
2.8 安全, 对电信行业安全的支撑程度
2.9 标准&规范遵循, 对电信行业标准&规范遵循程度，重点是TMF
2.10 Data & AI*, 重点分析，尤其是AI这一块的规划，能力等.
3. 商业&交付
3.1 定价&License
3.2 渠道情况,包括销售渠道，交付渠道，维护渠道等
3.3 实施&运维, 厂商直交付 vs SI 生态；实施工期与人力依赖; SLA、版本节奏、补丁策略、长期支持（LTS），交付假设条件
3.4 典型客户CASE, 包括帮助客户解决了什么问题，带来了什么价值
4. 总结
4.1 SWOT分析.  现阶段先不做.
5. 附录
5.1 引用资料 这个章节填写形成竞品分析报告引用的资料和来源. 如果是引用的本地文件，请填写对应的文件名。如果是引用信息来自互联网，请填写对应的URL。

# 内容与撰写要求

- **有材料则写透**：凡抽取到的 text 中有与某章节相关的信息，应**充分引用并展开**，写出具体要点、数据、案例或对比，避免整节只有一两句概括。可多用列表、小标题或表格呈现，使内容可读、可落地。
- **无材料才 TODO**：仅当某章节在输入中确实找不到对应材料时，才用 TODO 占位；有材料但未写实、写透，不算达标。
- **避免干涩**：不要只做「有/无」「支持/不支持」的罗列；尽量写出「是什么、覆盖哪些、与竞品或行业对比如何、典型场景或客户怎么说」等，使报告具有分析性和参考价值。
- **数据与出处**：能从原文提取的数字、项目名、版本号、客户名等应尽量保留并写入正文，便于溯源；必要时在附录引用资料中注明来源。

## 各章节撰写要点（示例性要求）

以下为易写短、写干的章节给出的示例性要求，撰写时尽量达到或参考；有材料的章节应至少满足对应要点，无材料处用 TODO。

| 章节 | 示例性要求 |
|------|------------|
| 1.1 行业位置&愿景 | 写出公司定位、主要业务线、目标客户与地域；若有愿景/战略表述，摘录 1～2 句并简要解释。 |
| 1.2 规模&营收 | 尽量给出人员规模、营收区间或典型项目金额；若原文无具体数字，写明「据公开/受访信息」等出处。 |
| 1.3 市场情况 | 分区域或分客户类型简述市场布局；含典型客户名、项目名或区域名单，避免仅一句概括。 |
| 1.4 产品组合 | 列出主要产品/方案名称及对应领域（如资源/开通/保障）；可配简短列表或与竞品对照。 |
| 1.5 组织架构 | 写出与报告相关的组织设置（如事业部、片区、交付模式）；若有规模或汇报关系，一并写出。 |
| 2.1 架构&技术分析 | 写出技术栈、部署方式、高可用/扩展方式；若有架构分层或版本演进，分点或分段说明。 |
| 2.2 功能分析 | **必须分开通、保障、资源三大块**；每块至少 2～3 个要点（如核心模块、关键能力、与标书/规范的对应）；可列表或小表。 |
| 2.3 业务支撑能力 | 按业务类型（个人/家客/政企等）或生命周期（规划—建设—运维）写出支撑点；有案例或客户名可写出。 |
| 2.4 网络支撑能力 | 按网络专业（无线/核心/传输/IP/5G/NFV 等）写出覆盖与协议；若有缺失项，明确写出「未覆盖」或 TODO。 |
| 2.5 系统集成能力 | 写出 OOTB 适配器、协议、对接方式；被第三方集成的 API 或开放能力若有材料则写出。 |
| 2.6 可配置性 | 针对服务目录、流程模板、资源策略、激活/指令、施工项等，逐项写出可配置程度或示例；避免仅「支持/不支持」一句。 |
| 2.7 开放性 | 写出二开方式、低码/零码支持情况；若有能力开放或生态描述，摘录要点。 |
| 2.8 安全 | 写出与电信/行业安全相关的能力或认证；无材料则 TODO。 |
| 2.9 标准&规范遵循 | 列出遵循的标准/规范（如 TMF eTOM、TMN、3GPP、ITIL）；若有对标或认证，写出。 |
| 2.10 Data & AI | 写出数据能力、AI 应用场景或规划；若有产品名、版本或路线图表述，摘录。 |
| 3.1 定价&License | 写出定价模式、典型价格区间或 License 类型；若有项目金额区间，写出。 |
| 3.2 渠道情况 | 分销售、交付、维护写出渠道类型与代表案例；若有合作伙伴名，写出。 |
| 3.3 实施&运维 | 写出交付方式（直交付/SI）、典型工期与人力、SLA/版本/补丁/LTS 等若有则写出。 |
| 3.4 典型客户 CASE | 每个案例写出客户名、项目/场景、解决的问题与带来的价值；至少 2～3 个案例，每案 2～4 句。 |

# 文档风格定义如下

以下约定用于生成「竞品分析报告」docx 时的版式与样式，使输出文档统一、易读、便于打印与归档。

---

## 1. 页面与版心

| 项目 | 设定 | 说明 |
|------|------|------|
| 纸张 | A4 | 210mm × 297mm |
| 页边距 | 上下 2.54cm，左右 3.17cm | 与 Word 默认「普通」一致，便于批注与装订 |
| 方向 | 纵向 | 图表过宽时可单独设横向节 |
| 页眉 | 可选 | 居中或外侧：报告简称 + 版本/日期，小五号、灰色 |
| 页脚 | 建议 | 居中：页码（如「第 N 页 共 M 页」或纯数字），小五号 |

---

## 2. 字体与字号

| 用途 | 中文字体 | 西文字体 | 字号 | 字重/其他 |
|------|----------|----------|------|-----------|
| 封面标题 | 微软雅黑 Bold | Arial / Calibri Bold | 22–26pt | 居中，可加 1.5 倍行距 |
| 一级标题（章） | 微软雅黑 Bold | Arial Bold | 16–18pt | 如：1. 公司概况、2. 产品分析 |
| 二级标题（节） | 微软雅黑 Bold | Arial Bold | 14pt | 如：1.1 行业位置&愿景 |
| 三级标题（小节） | 微软雅黑 或 宋体 Bold | Calibri Bold | 12pt | 若 2.2 下再分「开通/保障/资源」等 |
| 正文 | 微软雅黑 | Times New Roman / Calibri | 五号（10.5pt）或 小四（12pt） | 正文统一一种，全篇一致 |
| 图注/表注 | 微软雅黑 | 同上 | 小五（9pt） | 可选灰色，段前 6pt、段后 12pt |
| 引用/附录列表 | 宋体 | Times New Roman | 五号 | 悬挂缩进或左侧对齐 |

- **原则**：全篇不混用多套中文字体。
- **数字与英文**：与所在段落西文字体一致，避免在中文段落中出现明显字体跳跃。

---

## 3. 标题与段落

| 层级 | 样式名建议 | 段前 | 段后 | 行距 | 其他 |
|------|------------|------|------|------|------|
| 一级（章） | 标题 1 | 24–30pt | 12pt | 单倍或 1.15 倍 | 可加分页符：每章从新页开始（可选） |
| 二级（节） | 标题 2 | 12–18pt | 6pt | 单倍 | 与正文区分明显 |
| 三级（小节） | 标题 3 | 6–12pt | 3pt | 单倍 | 不单独成页 |
| 正文 | 正文 | 0 | 0 | 1.25–1.5 倍 | 首行缩进 2 字符（约 0.74cm） |
| 图/表注 | 题注 | 6pt | 12pt | 单倍 | 居中或左对齐（与图/表一致） |

- **编号**：一级用「1. 2. 3.」，二级用「1.1 1.2」，三级用「1.1.1」，与大纲一致；编号与标题文字之间留一空格。
- **分页**：若希望每章从新页开始，在一级标题前插入分页符。

---

## 4. 列表与表格

- **列表**
  - 无序：采用「●」或「–」，与正文同字号，悬挂缩进 1 字符。
  - 有序：1）2）或 1. 2.，与正文同字号。
  - 多级列表：缩进层级清晰，同一层级符号一致。
- **表格**
  - 表头：黑体或微软雅黑 Bold，小五或五号，底纹浅灰（如 15% 灰）。
  - 表格线：外框 0.5pt，内部 0.25pt；或仅保留横线（三线表）。
  - 表题：在表上方，格式同「图注/表注」，编号如「表 2-1 功能模块对比」。

---

## 5. 图与图注

- **图片**：居中；宽度不超过版心（约 14–16cm），按比例缩放，不拉伸变形。
- **图注**：置于图下方，格式同「图注/表注」，编号如「图 1-1 产品版图」「图 2-1 系统架构」。
- **引用**：正文中首次出现时写「如图 1-1 所示」等，与图注编号一致。

---

## 6. 颜色与强调

| 用途 | 建议 | 说明 |
|------|------|------|
| 正文 | 黑色 | 默认 |
| 一级/二级标题 | 黑色 或 深灰（如 RGB 40,40,40） | 打印友好 |
| 超链接 | 蓝色 + 下划线 | 若保留 URL |
| 强调 | 加粗，不推荐大面积彩色 | 少量关键词可加粗 |
| 表格/图注 | 黑色 或 深灰 | 可选灰色 50% |

- **原则**：以黑白灰为主，颜色仅作少量点缀，保证打印和复印清晰。

---

## 7. 封面与目录

- **封面**（必选，单独一页）：
  - 封面占一整页，与正文分页。
  - **主标题**：`<竞品名>竞品分析`（如「浪潮竞品分析」），居中，字体字号见 §2 封面标题（黑体或微软雅黑 Bold，22–26pt），可加 1.5 倍行距。
  - **副标题**：包含 **分析人**、**分析时间**；可另起行写版本号（如 v1.0）、报告日期等。副标题居中，字号小于主标题（如小四或五号），分析时间建议格式 YYYY-MM-DD。
  - 排版：主标题、副标题均在版心内居中；封面页不编正文页码（若需要可单独设封面页码样式或不显示）。
- **目录**（可选）：使用 Word 自动目录（基于标题 1–3），目录页单独一页，标题为「目录」，二级样式；更新域后保持页码与正文一致。

---

## 8. 附录与引用

- **5. 附录**下「5.1 引用资料」：按类型分段（如「本地文件」「网页」），每条一条目，可加编号或项目符号。
- 本地文件写清文件名；来自网页的写清 URL；若有作者/发布机构可一并注明，便于溯源。

---

## 9. 实施说明（供脚本/模板使用）

- 生成 docx 时建议：先应用上述字体与段落样式，再写入内容；一级标题使用「标题 1」、二级使用「标题 2」，正文使用「正文」并设首行缩进与行距。
- 若使用 python-docx：在写入段落前设置 `paragraph.paragraph_format`（段前段后、行距、首行缩进）与 `run.font`（字体、字号、加粗），或先定义 `document.styles` 再引用样式名。
- 图片插入后单独插入一段作为图注，并应用题注样式（小五、灰色、段前段后间距）。
